`pixel-aquarium` は、ブラウザ全体をひとつの水槽に見立てて、横から眺めるドット絵アクアリウムを体験できる1ページアプリである。

# 1. 全体方針・コミュニケーション

- ユーザーは日本人です。コード内コメント・最終出力メッセージ・ユーザーへの質問は日本語でお願いします。
- 既存のコードコメントは、明示的な指示がない限り変更しない。
- `src/scripts` 以下の TypeScript コードを実行するときは `node --import tsx ./src/scripts/hello.ts`
- 実装中に必要なnpmライブラリあれば許可なしでインストールしてよい

ライブラリ概要

- 言語: TypeScript
- UI: React v19 / Mantine v9
- hook: Mantine の hook を使用
- Lint: biome v2
- ルーティング: react-router-dom（`BrowserRouter` + `Routes` + `Route`）を使用。ルート定義は `src/App.tsx`
- ビルドはvite v8

# 2. pixel-aquariumについて

## このプロジェクトの目的

このプロジェクトが目指すのは、現実には魚を飼いにくい大人に対して、次の感覚を与えること。

- 開いた瞬間に、水槽の中に入ったように感じられる
- 何もしなくても、眺めているだけで落ち着ける
- 少し触ると、水や魚から反応が返ってきて満足感がある

単なるレトロゲーム風演出ではなく、「所有の代わりになる鑑賞体験」を作ることを優先する。

## 体験の中核

- 画面全体がひとつの水槽として成立していること
- ドット絵らしい輪郭の気持ちよさがあること
- ただし動きは硬すぎず、生き物と水の気配が感じられること
- 放置して眺める時間と、軽く触って反応を見る時間の両方が成立すること

実装判断で迷ったら、「情報量を増やすこと」より「水槽として自然に感じること」を優先する。

## 現時点の確定仕様

- Pixi.js を使う
- ビジュアルはピクセルアート、ドット絵で統一する
- まずは1ページで完結するデモとして作る
- 魚は常時泳いでいる
- 魚は複数種類とし、速度・泳ぐ深さ・性格に差を持たせる
- 魚の見た目は、明るい色の熱帯魚を中心に、形や配色のバリエーションを持たせる
- 魚種はクマノミ、ナンヨウハギ、キイロハギ、ネオンテトラ、ベタ、紅白の錦鯉風などを基準に、多様性が感じられる構成にする
- 視点は常に水槽を横から見る固定構図とする
- 1画面の中に、水面から砂地までを収める
- 画面上部が水面で、画面全体のうちおよそ 9 割が水中として見える構成にする

## 見た目の方向性

- ドット絵の輪郭はくっきり保つ
- 拡大時もピクセルのエッジがにじまず、輪郭の気持ちよさが失われないことを重視する
- 尾びれや姿勢の小さな変化で、魚が静止画ではなく泳いでいると感じられること
- 水面、水草、泡、光はそれぞれ別のリズムで動き、画面全体が単調にならないこと

## 水槽内に必要な要素

- 水草のゆらぎ
- 岩
- 砂
- 泡
- 光のゆらぎ

これらは装飾ではなく、水槽らしさを成立させるための必須要素として扱う。

## インタラクション仕様

- クリックまたはタップで波紋が出る
- 波紋や接触に反応して魚が逃げる
- 水面より上をクリックまたはタップすると、餌が水面下へ落ちる
- 餌が落ちると、魚がその餌に寄ってくる

操作は少なくてよいが、入力に対して「水」と「魚」の両方が反応している感覚を大事にする。

## 現在のデモ到達点

- 8匹の魚が常時遊泳し、種類差に加えて個体差も感じられる状態を目指す
- 魚ごとに速度、遊泳深度、臆病さ、食欲、旋回性能の差を持たせる
- 水面は上部 1 割の帯として認識でき、水中との境界が見た目でも明確であること
- 水草のゆらぎ、岩、砂地、泡、光のゆらぎが同時に存在し、静止画ではない水槽感を作る
- 水中タップで魚が逃げ、水面上タップで餌に寄るという2種類の反応が成立していること
- 画面サイズが変わっても、水槽の構図と体験が崩れないこと

## 非目標

- 音は入れない
- 昼夜や時間変化は入れない
- UI を増やしすぎない
- ゲーム化やスコア化を主目的にしない

## 将来拡張

- 魚数の調整 UI
- 反応強度の調整 UI

ただし現段階では、拡張性よりも「最初の水槽体験が成立しているか」を優先する。

## 実装・表現の指針

- ドット絵の輪郭は崩さない
- その一方で、動きは滑らかにし、古いゲーム画面ではなく生きた水槽に見せる
- 反応を増やしすぎて騒がしくするより、静かに眺められる密度を保つ
- 画面要素を追加する場合は、鑑賞体験を強めるかどうかで採否を判断する


# 3. TypeScript / コーディングスタイル

- 型定義は `interface` ではなく 必ず `type` を使う。
- `any` 型は 絶対に使用しない。
- 型アサーション `as` は原則使用しない。やむを得ず `as` を使う場合は、なぜ必要かをコードコメントで説明すること。
- `class` 構文は 一切使用しない。
- 関数定義は すべてアロー関数 を使用する。
- 条件分岐は 早期リターンを用いてフラットに保つ。
- `try-catch` は乱用せず、必要最低限のみ使用する。
